# MClash 1.1.8

This patch release fixes the signed Network Extension control path and makes DNS Routing part of the default App Routing lifecycle.

## App Routing startup fixed

- Developer ID signatures now include the application and team identities macOS requires when the host communicates with the transparent proxy Provider.
- Release validation compares those identities across both provisioning profiles, the declared entitlements, and the final signed Host and System Extension. A mismatched package now fails before publication.
- Network Extension activation, Provider verification, DNS activation, rollback, and shutdown now write durable Unified Log events with the operation and revision that failed.

## DNS starts and stops with App Routing

- DNS Routing is enabled by default and starts and stops in the same transaction as App Routing.
- Existing 1.1.7 settings are migrated so the old `dnsEnabled=false` default cannot silently leave DNS outside MClash after upgrading.
- The ordinary DNS switch has moved into a collapsed Advanced DNS Routing section for the exceptional case where another DNS Proxy must remain active.
- Enabling App Routing explains that macOS permits one active DNS Proxy and that MClash can replace Proxifier DNS or another provider.
- If DNS health becomes unverified while running, MClash stops DNS and application capture together and restores macOS system DNS instead of leaving a partially active routing mode.

## Verifiable DNS state

- MClash now reloads and verifies the system-owned `NEDNSProxyManager` configuration after saving it, including ownership, enabled state, revision, and activation identity.
- Runtime health checks confirm both the persisted macOS configuration and the Provider heartbeat, so an old heartbeat cannot be mistaken for an active DNS Proxy.
- Terminal Provider startup failures are reported immediately, and DNS preference errors retain their Network Extension error domain and stage for diagnosis.
